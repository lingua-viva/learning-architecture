## EXP09 — explicit before/after

**Before**: `static/index.html`'s `renderHealth()` used `status.data === "OK" ? "ok" : "warn"` — every non-OK status (including `PRIVATE_RISK`, a privacy stop) rendered with the identical `.badge.warn` styling as a routine `WARN`/`FIXABLE`/`UPDATE_AVAILABLE`/`BLOCKED`. A teacher had no visual signal that a privacy issue was more urgent than "something may need attention."

**After**: a `healthBadgeClass(status)` helper maps `OK → "ok"`, `PRIVATE_RISK → "risk"`, everything else → `"warn"`. A new `.badge.risk` CSS rule (`--risk: #a3222c`, filled red background, white text, `font-weight: 600`) gives `PRIVATE_RISK` visually distinct, more urgent styling than the routine amber `.badge.warn`.

**Live repro used to confirm the bug was real** (not hypothetical) before fixing:
```

curl -s http://127.0.0.1:8787/api/health | python3 -m json.tool
# -> {"status": "PRIVATE_RISK", ...} — confirmed reachable via the real
#    Doctor check_privacy_paths(), not a hypothetical status value.
rm observations_repro_test.txt
```
